## Находка 1 — Неизвестный макрос (unknownMacro)

### Идентификатор и критичность

| Поле        | Значение                          |
|-------------|-----------------------------------|
| **Правило** | `unknownMacro`                    |
| **Уровень** | `error`                           |
| **Файл**    | `libtiff/tif_aux.c`, строка 114, столбец 28 |

### Сообщение Cppcheck

```
/analysis/libtiff-src/libtiff/tif_aux.c:114:28: error:
There is an unknown macro here somewhere. Configuration is required.
If TIFF_SSIZE_FORMAT is a macro then please configure it. [unknownMacro]
```

### Описание участка кода

`tif_aux.c` содержит вспомогательные функции libtiff. На строке 114 используется макрос `TIFF_SSIZE_FORMAT` — это format-спецификатор для типа `tmsize_t` (знаковый аналог `size_t`). Макрос определяется в `tiffconf.h`, который генерируется системой сборки (`./configure` или CMake) на основе характеристик платформы.

```c
/* tif_aux.c ~строка 114 */
TIFFErrorExt(tif, module,
    "%" TIFF_SSIZE_FORMAT " byte request exceeds ...", size);
```

### Оценка валидности

**⚠️ Не подтверждается — артефакт анализа**

Cppcheck анализировал исходники без предварительной сборки (`./configure`), поэтому `tiffconf.h` не был сгенерирован и макрос `TIFF_SSIZE_FORMAT` остался неизвестным. В реально собранном проекте этот макрос определён корректно (например, `"%zd"` на Linux x86_64). Это ложное срабатывание, вызванное отсутствием конфигурации при статическом анализе.

Аналогичная сработка присутствует в 20+ файлах проекта по той же причине.

### Потенциальный риск

Реального риска нет — это артефакт инструмента. Однако если бы макрос действительно отсутствовал, это привело бы к ошибке компиляции.

### Рекомендация

Запускать Cppcheck после выполнения `./configure` или передавать определения макросов явно:
```bash
cppcheck -DTIFF_SSIZE_FORMAT='"%zd"' -DTIFF_SIZE_FORMAT='"%zu"' ...
```
Или использовать `compile_commands.json` (CMake): `cppcheck --project=compile_commands.json`.

## Находка 2 — Синтаксическая ошибка (syntaxError)

### Идентификатор и критичность

| Поле        | Значение                          |
|-------------|-----------------------------------|
| **Правило** | `syntaxError`                     |
| **Уровень** | `error`                           |
| **Файл**    | `libtiff/tiff.h`, строка 79, столбец 41 |
| **Файл**    | `libtiff/tiffio.h`, строки 590–600 |

### Сообщение Cppcheck

```
/analysis/libtiff-src/libtiff/tiff.h:79:41: error: syntax error [syntaxError]
/analysis/libtiff-src/libtiff/tiffio.h:590:17: error: syntax error [syntaxError]
... (ещё 10 строк в tiffio.h)
```

### Описание участка кода

`tiff.h` — главный заголовочный файл libtiff, определяет базовые типы. Строка ~79 содержит условную компиляцию с использованием нестандартных типов или атрибутов, которые зависят от предварительно определённых макросов платформы:

```c
/* tiff.h ~строка 79 */
#if defined(__WIN32__) || defined(_WIN32)
typedef UINT16 uint16_t;   /* требует <windows.h> или stdint.h */
#endif
```

`tiffio.h` строки 590–600 — объявления функций с атрибутами типа `__attribute__((format(...)))` или аналогичными, которые Cppcheck не может разобрать без полного контекста препроцессора.

### Оценка валидности

**⚠️ Не подтверждается — артефакт анализа**

Реальной синтаксической ошибки в коде нет — libtiff 4.4.0 успешно компилируется на всех поддерживаемых платформах. Cppcheck споткнулся о платформо-зависимые конструкции, которые раскрываются только после препроцессора. Без `#include` системных заголовков и без определений платформенных макросов парсер Cppcheck видит «синтаксическую ошибку» там, где её нет.

### Потенциальный риск

Реального риска нет. Множество сработок `syntaxError` в одном файле — признак того, что анализатору не хватает контекста препроцессора.

### Рекомендация

Добавить системные include-пути и подавить ложные срабатывания:
```bash
cppcheck --suppress=syntaxError \
          -I /usr/include \
          -I /analysis/libtiff-src/libtiff \
          ...
```
Либо использовать `--project=compile_commands.json` для полного контекста сборки.

## Находка 3 — Утечка памяти при realloc (memleakOnRealloc)

### Идентификатор и критичность

| Поле        | Значение                          |
|-------------|-----------------------------------|
| **Правило** | `memleakOnRealloc`                |
| **CWE**     | CWE-401 (Memory Leak)             |
| **Уровень** | `error`                           |
| **Файл**    | `tools/fax2ps.c`, строка 367      |

### Сообщение Cppcheck

```
/analysis/libtiff-src/tools/fax2ps.c:367:21: error:
Common realloc mistake: 'pages' nulled but not freed upon failure [memleakOnRealloc]
```

### Описание участка кода

`fax2ps.c` — утилита конвертации TIFF/FAX в PostScript. На строке ~367 выполняется расширение динамического массива `pages` через `realloc`:

```c
/* fax2ps.c ~строка 367 */
pages = realloc(pages, (nPages + 1) * sizeof(pages[0]));
if (pages == NULL) {
    /* ОШИБКА: при неудаче realloc вернул NULL,
       но оригинальный указатель 'pages' уже перезаписан —
       старая память недоступна и не может быть освобождена */
    fprintf(stderr, "Out of memory\n");
    return;
}
```

Классическая ошибка: `realloc` при неудаче возвращает `NULL`, но не освобождает старый блок памяти. Если результат записывается напрямую в тот же указатель, при ошибке старый указатель теряется — происходит утечка памяти.

Аналогичная ошибка найдена в `tools/tiffdump.c:260` (`visited_diroff`).

### Оценка валидности

**✅ Подтверждается**

Это реальная и распространённая ошибка в Си-коде. Паттерн `ptr = realloc(ptr, new_size)` — классический источник утечки памяти, задокументированный в стандартах безопасного программирования (CERT C MEM04-C). Cppcheck корректно её идентифицировал.

### Потенциальный риск

- **Утечка памяти:** при нехватке памяти старый буфер `pages` не освобождается.
- **В долгоживущих процессах** (серверы обработки документов) многократные неудачные `realloc` приводят к постепенному исчерпанию памяти.
- В данном случае (`fax2ps` — утилита командной строки) практический риск невысок, но паттерн опасен в библиотечном коде.

### Рекомендация

Использовать временный указатель:

```c
/* Безопасный паттерн realloc */
void *tmp = realloc(pages, (nPages + 1) * sizeof(pages[0]));
if (tmp == NULL) {
    fprintf(stderr, "Out of memory\n");
    free(pages);   /* освободить старый блок */
    pages = NULL;
    return;
}
pages = tmp;
```
